# db-postgres

nifra as a **pure backend** with **Drizzle ORM + Postgres** — a typed todos API with real migrations.

It uses **PGlite** (Postgres compiled to WASM, embedded in-process) so it runs with zero setup — no
Docker, no `psql`. PGlite *is* Postgres, so the schema, queries, and migrations are exactly what you'd
run against a server. For production you change three lines in `db.ts` (shown there) to point Drizzle at
`postgres(process.env.DATABASE_URL)` — nothing else moves.

```sh
bun run examples/db-postgres/server.ts
curl localhost:3000/todos
curl -X POST localhost:3000/todos -H 'content-type: application/json' -d '{"text":"ship it"}'
```

- `schema.ts` — the Drizzle schema (the source of truth: `TIMESTAMPTZ`, `NOT NULL`, serial PK).
- `migrations/` — SQL generated by `drizzle-kit generate` from the schema; applied by Drizzle's migrator on startup.
- `db.ts` — the connection + migrate step (swap PGlite → `postgres-js` for a server).
- `backend.ts` — the `server()` routes; `POST /todos` validates `text` before any query, and Drizzle parameterizes every value.

**Workflow:** edit `schema.ts` → `bunx drizzle-kit generate` (writes a new migration) → restart. The
same flow targets a real Postgres, Neon, Supabase, or PlanetScale-for-Postgres by changing only the
connection string.
